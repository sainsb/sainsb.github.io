The Teensy and the 4-Digit 7-Segment display

I ordered a couple of 4-Digit 7-Segment displays from Sparkfun the other day. Not entirely sure what I’ll do with them but they have been fun to familiarize myself with. I began by following some blogged instructions at AllAboutEE. While there are some valuable gems there, the setup is not correct and has resistors inline with the digit pins instead of the segment pins. In the end, I got more out of an old Arduino Forum post describing the method of connecting this to the Arduino.

Here is my horrible Fritzing diagram:

And a slightly less horrible video:

Source Code: (Note the use of the Teensy PORTB - it made for horrible wiring but simple and efficient coding.)

int digit0 = 7; 
byte numbers[10] =
{
  B11000000, // 0
  B11111001, // 1
  B10100100, // 2
  B10110000, // 3
  B10011001, // 4
  B10010010, // 5
  B10000010, // 6
  B11111000, // 7
  B10000000, // 8
  B10010000  // 9
};

void setup()
{
  DDRB=0xff;
  PORTB=0x00;
  for (byte digit = 0; digit < 4; digit++)
    pinMode(digit0+digit,OUTPUT);
}

int perDigitTime = 100;
int digitPosition = 1;  
unsigned long previousMillis = 0;

int value = 0;
int powerOfTen[] = {1,10,100,1000};

void RefreshDisplay()
{
  digitalWrite(digit0 + digitPosition, LOW);
  digitPosition++;
  if (digitPosition > 3) digitPosition = 0;

  int digitValue = value % powerOfTen[ digitPosition+1 ] / powerOfTen[digitPosition];
  int number =  numbers[ digitValue ];

  if ((number > 0) || (value < powerOfTen[ digitPosition+1 ]))
  {
  PORTB=number;
  digitalWrite(digit0 + digitPosition, HIGH);
  delay(4);
  }
}

void loop()
{
  unsigned long currentMillis = millis();
  if (currentMillis - previousMillis > perDigitTime)
  {
    previousMillis = currentMillis;
    value++;
  }

  RefreshDisplay();
}

The rotary encoder.

I am pretty noob when it comes to embedded development. A few months back I received a blown up amplifier from my in-laws…I took it apart to discover a failed relay critical to switching on power…After googling for awhile it appeared to be a common problem with an expensive solution. Long story short, I bastardized the controls on the front of it in search of potentiometers for some learning I was doing at the time. What I pulled off were not potentiometers. I did not know what they were for the longest time and when I wired one up as an analog potentiometer to my Netduino…I got a rather disturbing BLINK on the board and decided to immediately stop and work on something else. ~6 months later I’ve learned a little more and decided to give this dial a go. Here’s the breakdown pulled directly from a sharp fellow’s site

Well the potentiometers have some disadvantage.

  • Produce noise at knob turn over the uses or if dust has passed in.
  • They are not that accurate.
  • To use them in a digital circuit you have to use an Analog To Digital converter.

On the other hand Rotary Encoder.

  • There is no noise production (if you use the appropriate capacitors).
  • They are accurate (they have steps).
  • There is no need of a digital to analog converter.

I’m not going to go in to the particulars of the device, the aforementioned link provides a nice little tour for beginners. Ultimately, I wired up my Teensy to my 10-LED bar. The action on encoders is very smooth with subtle bumps for feedback. The LEDs were pretty simple and so for extra credit I included some mouse and keyboard commands to adjust the volume on my box whilst using the excellent 3RVX.exe global volume controller for Windows. Hooking up input parameters in custom script tools

  

    

    

      
        
This functionality has been a mystery to me for awhile but I took the time to figure out what’s going on. (And now I’m sharing it with you (because sharing is caring))


Say you create a geoprocessing script with an input and then want to also specify fields within the input dataset as part of the task (Make sense? if not it will). I set up an input param of type ‘Raster Dataset’ and two input params of type ‘Field’ and foolishly expected the field inputs to automagically wire themselves up to the specified dataset. Turns out it takes a little more than that and there’s a few curve balls.


    

  • Use the ‘String’ type instead of ‘Field’ for the input params that you want to wire up to respond to your source. Like this:
    • 



Ok - next, click over to the Validation Tab on your script properties:




You’re going to be working with the updateMessages function. My first inclination was to set the input param to the describe(dataset).fields object. Negative…
Here’s the goods assuming one input dataset as the first param and two following input params that take there cue from the specified dataset:


class ToolValidator:
  “””Class for validating a tool’s parameter values and controlling
  the behavior of the tool’s dialog.”””


  def init(self):
    “””Setup arcpy and the list of tool parameters.”””
    import arcpy
    self.params = arcpy.GetParameterInfo()


  def initializeParameters(self):
    “””Refine the properties of a tool’s parameters.  This method is
    called when the tool is opened.”””
    return


  def updateParameters(self):
    “””Modify the values and properties of parameters before internal
    validation is performed.  This method is called whenever a parmater
    has been changed.”””
    ds = self.params[0].value
    if ds:
        self.params[1].filter.list = [str(field.name) for field in arcpy.Describe(ds).fields]
        self.params[2].filter.list = [str(field.name) for field in arcpy.Describe(ds).fields]
    return


  def updateMessages(self):
    “””Modify the messages created by internal validation for each tool
    parameter.  This method is called after internal validation.”””
    return

Converting MySQL geometry to Shapefile

  

    

    

      
        
I was recently faced with the task of converting MySQL Geometry  (stored in a remote db) as WKB to a shapefile to an ESRI File Geodatabase.
Since it’s a remote database stored with my web hosting provider access is rather limited (e.g no console access). The dataset is ~33,000 features with 20-some fields - calling a dump from the phpMyAdmin web interface works but returns WKB. I tried reading this stuff with Shapely, but it wasn’t having it…messed around with encodings to no avail.  And while one can export the geometry as text, my dump limits are thus severely limited to ~2000 records per export (phpMyAdmin has a 5mb limit per export file as I recall).  I used to suffer through this when time was less valuable and I was less competent but not no more.


I decided to extract the geometry separately and join the attribution in later.  This made coding the automated extraction a little easier.  So how did I do it?  I went out and grabbed the MySQL Connector for .NET, threw in my credz and went to town 5 at a time so as not to stress the server.


using System;
using System.Collections.Generic;
using System.Linq;
using System.Text;
using MySql.Data.MySqlClient;
using System.Web;
using System.IO;
using System.Threading;

namespace myBitchinMySQLExtractor
{
    class Program
    {
        const string MyConString = "connectionStuff";
        const int numRecords = 35000;
        static void Main(string[] args)
        {
            for (int t = 0; t < numRecords; t += 5)
            {
                try
                {
                    getData(t);
                }
                catch (Exception ex)
                {
                    try
                    {
                        Thread.Sleep(10000);
                        getData(t);
                    }
                    catch (Exception exc)
                    {
                        Console.Write("Lost connection at " + t);
                    }
                }
                Thread.Sleep(3000);
            }
        }

        private static void getData(int t)
        {
            var connection = new MySqlConnection(MyConString);
            MySqlCommand command = connection.CreateCommand();
            MySqlDataReader Reader;
            connection.Open();
            command.CommandText = "SELECT ID, astext(geometry)"+
                "as geom from table where 1 limit " + t + ",5;";

            Reader = command.ExecuteReader();
            while (Reader.Read())
            {
                string thisrow = "";
                for (int i = 0; i < Reader.FieldCount; i++)
                {
                    var s = Reader.GetValue(i);
                    if (s.GetType() != typeof(byte[]))
                    {
                        thisrow += s.ToString() + ", ";
                    }
                    else
                    {
                        thisrow += Encoding.UTF8.GetString((byte[])s, 0, ((byte[])s).Length);
                    }
                }
                using (var sw = new StreamWriter("C:/myFile.txt", true))
                {
                    sw.WriteLine(thisrow);
                }
                Console.WriteLine(thisrow + Environment.NewLine + Environment.NewLine);
            }
            connection.Close();
        }
    }
}




Ok, so now I had a big-ass file full of ids and geometry in WKT.  What next? Shapely and GeoJSON to the rescue. 


from shapely.wkt import loads
from geojson import dumps
import re
import sys
import string

output=r"C:/myOutput.txt"
input=r"C:/myInput.txt"
f = open(output, "a")
f.write("{ \"type\": \"FeatureCollection\",\"features\": [")

def processWKT(id, line):
    x = loads(line)
    y = dumps(x)
    f.write("{\"type\": \"Feature\",\"geometry\": " +
                y + ", \"properties\" : {\"id\" :\"" + id+"\"}},\n")

file = open(input, "r")

for line in file.read().split("\n"):
    #parse the id out
    #grab everything up to the first comma
    s=re.match("[^,]*,\s",line)
    t=s.group()
    l = string.replace(line,t, "")
    y = string.replace(t,", ","")

    processWKT(y,l)
    print y

f.write("]}")
file.close()
f.close()




Ok, so now after I go in and delete a comma in the last line, I have a valid geojson file with all my IDs and geometry.  What next? GDAL.  Converting the geojson to a shapefile is a one-liner with ogr2ogr:


ogr2ogr -f "ESRI Shapefile" c:/myshapes.shp c:/myOutput.js OGRGeoJSON




So, yes, it is a painful process but these three steps helped to expedite the extraction of data from a remote MySQL db, convert it to the middle-ground format of GeoJSON, and on to a shapefile using GDAL.  All, with free and open-source libraries and platforms. Since my data comprises fields of > 256 characters, I’ll convert the shapefile to a file geodatabase and then populate the attribution using ESRI tools.

Weekend notes on the physical.

  

    

    

      
        
The weather being such fucking shit here lately it’s given me a lot of time to devote to my growing electronics hobby. While voltage and amperage seem like simple concepts, I have to admit, even with the water analogies, the concept of amperage has, for the most part, eluded me. A few things that clicked this weekend were:


    

  • Amperage is a rate. not a force.
    • 

  • Amperage is “the same at all points in a similar circuit.  It has to be, because the flow of electrons has no place to go” (Platt, 22).
    • 

  • Car batteries are designed so as to sustain massive draw.  Hundreds of amps may be required to spin the flywheel.
    • 

  • Current generates a magnetic force. Hence the I in Ohm’s Law (V=I*R) refers to inductance.
    • 

  • Conversely, a magnetic force can generate current. You can actually light up an LED by passing a high-powered magnet through the center of a coil.
