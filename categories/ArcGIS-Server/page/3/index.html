NAIP Server-side tiling

As a clearer follow up to this blogpost:

Problem A: MrSID tiled imagery is only realistically viewable at scales of < 1:50,000.

Problem B: In order to optimize viewing of imagery for the web it is best to tile imagery according to pre-set scales (e.g. the Google Maps/ESRI/Virtual Earth tiling scheme) - this is a rather time and space consuming chore. It also mandates that the imagery layer be considered an ArcGISTiledMapService for the intents and purposes of ESRI web-APIs

Solution: Server-side tiling.  In order to optimize NAIP imagery viewing:

  1. Create an empty GDB
  2. Create 4 empty unmanaged raster catalogs labeled L1, L2, L3, L4
  3. Created 3 fishnets at the same extent as the county bounds. L1 = 2x2, L2=4x4, and L3=8x8.
  4. Ran a script (see below) to create bookmarks out of features
  5. Exported JPGs based on bookmarks (this is kind of like a quick and dirty DSMapBook)
  6. Imported JPGs into each of the above corresponding Raster Catalogs
  7. For L4 I imported the sid files
  8. Created an MXD with 4 groups corresponding to the fishnet/raster catalogs and set their scales accordingly

End result - a dynamic map service that should draw equally fast at all scales and will allow for custom scales.

Server-side tiling

Many, if not all, of our imagery layers are tiled into MrSIDS, this makes display at large scales relatively fast in ArcMap.  However, even though these tiles are pyramided, viewing the layer at small scales is nearly impossible.  In ArcMap we can set the layer to simply display polygons at scales smaller than x but what about on the web?  We are pretty much forced to set the layer to be scale dependent and a fairly large scale at that.  In order to enable and optimize the viewing of these imagery datasets on the web (and perhaps in ArcMap), I’ve set up a handy suite of scripts to create pyramid style grids (3 levels) and tile imagery accordingly.  The cost is about another gig but from my perspective, the ability to view the map at scales smaller than 1:50K and down and the performance thereof, is worth every kb.

Point density based tiling.

In the continuing thoughts about tiling vectors, I’ve realized that, especially where contours are concerned, it doesn’t necessarily make the most sense to tile a straight fishnet-based grid.  This thought occurred to me this morning when a company called interested in obtaining paper maps for all the parcels in the county.  In the rural areas of the county, using sections as a standard grid cell makes sense but in areas with higher density, 8.5x11 will not cut it.  This grid could be produced by generating parcel centroid points, generating a grid and then running a point count per grid cell.  In cells with a point count greater than a certain threshold, the grid cell would be quartered again.

I ran this train of thought out to contours as well and it, again, makes sense.  In areas with a greater range of vertical relief, say the Grand Canyon, point density is going to be much, much higher.  To optimize tiling, or rather to equalize, tiles, it makes sense to take this into consideration.  More on this in future posts.

Tiled Vectors and ArcMap

Special thanks to Brian and MS Research for enlightening me on this one

Tiling is not only for raster data but can also be used to improve the performance of vector data in web and desktop GIS. If you’re still interested, read on…

As you all most likely recall I recently created a county-wide 20 ft. contour feature class. By all of my measures this new layer I had created should draw much faster than the existing 20ft contour layer (AgTopo20). I ran summary statistics on number of vertices for a comparative sample and came up with an exponentially greater number of vertices in the existing feature class…So how was it drawing so much faster than my new contours layer???

The other obvious difference I noticed between the two was that the existing FC (AgTopo20) has roughly twice the number of features and 1/3 less coverage.

hmmm….As mentioned above, Brian clued me in to the fact that even though ArcMap only draws the features in the viewing area, if the extent of the features drawn spreads beyond the viewing area, it has to draw that also.  Aha!

Continuing along this line of thought…I had been wondering why the public lands layer was drawing so slowly in the web app.  The polygons weren’t that complicated.  but again, even if the user was centered over Prescott with just a corner of a polygon in view, the engine would be drawing said polygon up near Ash Fork!!!  Checked the Public Lands layer in SDE (AgLandUseWk) and sure enough: 10 total features.

Discovered a new tool in Data Management:Features- Multipart to Singlepart. Ran this tool on the Public Lands FC and resulted in 1400 records.  Sure enough, draw time in ArcMap was much improved.

Several years ago I experimented with tiling vectors for improving vector display in the Virtual Earth (now Bing) mapping API. See this link for more on this concept.  I never thought then that this same principle could be applied to improve draw times in ArcMap.

If you have a county-wide dataset that is displaying EQUALLY AS FAST(SLOW) AT LARGE SCALES AS SMALL SCALES - think about exploding or tiling your features.  If exploding doesn’t work, follow this workflow to tile your vectors:

  1. Create a fishnet (Data Management::Feature Class) a 10x10 worked for me for the contours. (Yes, 100 tiles)
  2. Convert the ouput fishnet to polygons.
  3. Split the polygons into indiv. feature classes using a split by attribute tool
  4. Batch intersect your target feature class with the polygon feature classes
  5. Merge together intersect outputs.

There are probably other (perhaps better) ways to do this but this worked well for me.

Moral of the story: Tiling is not only for raster data but can also be used to improve the performance of vector data in web and desktop GIS.

UPDATE - The Public Lands Layer was drawing pretty well on the web but there was still a lag..why? Oh yeah, because the layer in the labels service was still pointing to the old Public Lands layer - re-pointed it and it sped up (although this doesn’t make entire sense since none of the features were being drawn.  It was probably due to the labeling engine needing to quasi-draw the features in order to know where to label.

Initial thoughts on tuning ArcGIS Server

Our production GIS web server died again today at 3:50.  I was not surprised.  I had been watching the load not dip below 98% for a sustained period (like an hour).  We rebooted.  Ops had already made their system modifications and I made the following modifications:

Set the maximum number of instances for each service from 2 to 1.

Set all services that are not supposed to be running (File_gdb test services and secured services) to not start automatically and stopped them.

Set the host/SOC capacity to 100 - According to ESRI - “capacity limits the number of service instances that can run on a container machine (SOC)” - I’m not really sure how this translates  - in a way it doesn’t make sense since if you only have 10 services and have set their maximum number of instances to 1 each then it seems to me you would never have more than 10 service instances.?? Am I missing something? I have a hunch that within each ArcSOC instance on the web server there are multiple instances running. This is the only way the capacity number makes sense to me…

If/when the server begins to overheat again we will try the following additional settings:

Setting the services to run in low isolation - according to ESRI - “the isolation level determines whether these instances run in separate processes or share processes” - low isolation implies that service instances will share processes and according to ESRI “can significantly improve memory consumption on your server” but is risky in that “if a process experiences a shutdown or crash” (I don’t know why one would) “all instances sharing the process will be destroyed.”

ALSO:I noticed that the lsass.exe process is coming out as a prime suspect in overheating Mapserver. After googling ArcGIS Server and lsass.exe voila! arrived at this ESRI kb article outlining the correlation, or should I say causation.  It seems that every REST and SOAP request require the use of the LSASS (Local Security Authority Subsystem Service). By creating a new app pool and invoking additional IIS incantations, you can “disable per request authentication.”
