On open source vs. proprietary

Obviously this is a big, multifaceted issue, I just wanted to get this micro-thought down before it slips away.

In an organization where turnover is high, for whatever reason, it is obviously best to choose turnkey, proprietary software solutions that you can guarantee the hiring pool has experience with. This decreases labor overhead but likely increases software licensing overhead.

In an organization where turnover is low, for whatever reason, it might make more sense to allow for custom open source development. This increases labor overhead but likely decreases software overhead.

C# SOE vs. Python GPTool

Just watched an excellent video comparing the performance of C# Server Object Extensions(SOE) to Python geoprocessing services. I really like the tack he takes here: yes, the C# SOE is roughly twice as fast, but there are several other factors that should be taken into account.
For instance:

  • Programmer productivity - The core Python script (using arcpy) is only a few lines of code. Wrangling the input params and the response object is almost totally abstracted.* Portability - While the python script requires a toolbox with a script (+ params) wrapper, it is relatively easier to publish than an SOE, which must be registered with the Server. Though to be fair, once you understand how to register the SOE, it only takes a second; most of the pain is in decoding the documentation on how to do it.

So now I see why Ismael Chivite advised devs to only build SOEs to when you need atomic functionality. Are you ready for ArcGIS Server? I’m not.

ArcGIS Server Performance tests

I had a few free moments this morning to run some simple performance tests on our QA server instance.

I specifically wanted to test pooling and isolation. So here ya go.

The test consisted of 200 asynchronous (simultaneous) query requests to a feature class stored locally in a file geodatabase. All tests were performed immediately after restarting the service to emulate the server in it’s ‘resting’ state. (That wasn’t intentional but rather incidental)…

The 5 rounds of testing included the service set at:

  1. a min. of 1 instance and a max of 2 instances under high isolation
  2. a min. of 1 instance and a max of 2 instances under low isolation(8 instances per process)
  3. a min. of 1 instance and a max of 10 instances under high isolation
  4. a min. of 5 instance and a max of 10 instances under high isolation
  5. a min. of 5 instance and a max of 10 instances under low isolation(8 instances per process)
     

Here were the performance signatures:

Observations:

1. Low isolation provides no benefit under this configuration. That is to say, isolation may make a difference under higher loads, differing data sources etc. From my testing, I saw no significant improvement

2.The min. 1, max 10 test displayed a steep climb at first as instances were (I’m assuming) created but performed just as well as the tests with 5 standing instances.

Conclusion

Therefore, leave your services under high isolation and don’t worry about minimum instances. I believe I saw a recommendation for 0 minimum instances in the forums. According to the testing this would prove accurate as it does not appear to be very expensive to fire up an instance/instances under load.

Server Object Extension notes

A few notes while this is fresh in the gray matter.

  • When registering a SOE, you can’t use the standard .NET regasm.exe, you have to use esriRegasm.exe which is commonly located at C:\Program Files (x86)\Common Files\ArcGIS\bin. When running this utility you’ll be prompted to choose the application that the library will be associated with (Server, Desktop, Engine).
  • Next thing to make sure when registering your SOE is that the SOM account has read access to the .dll…Note that I’m ~85% sure of this, I was making changes fast and read this elsewhere…seems harmless enough if incorrect
  • You can register multiple operations under one SOE - under the CreateRestSchema method where REST methods are registered, just add to the RestResource.operations collection.
  • The RestOperation constructor takes 4 basic params (name, input arg names, response formats, and handler method) with 2 overloads (required capabilities (for security), and a bool for post only)
  • Apparently it is possible to alter the MIME type of the response - this thread in the ESRI support forum is good stuff.
  • Debugging live SOEs is actually pretty straightforward. Once the SOE is registered, publish a service (SOEs can only be associated with MapServer services at this point)[weak], add the SOE functionality to the service, and attach to the process. I only had one service running on my dev box so it was rather easy to figure out which ArcSOC.exe proc to attach to. I’m assuming the SOE’s are marked by the T-SQl/Managed code attributes…

ArcGIS Server Find on tiled service

Often times when creating a tiled map service for vector features, you can link to the data source and allow for identify and find functionality at the REST endpoint.
Note that the ‘Query’ method is not available as this is specific to vector services. BUT the ‘Find’ function does give us some pretty potent access to the data, especially with the new ‘Layer Definitions’ field. A certain individual was interested in querying Multnomah County’s taxlot data based on sale date…


Note that in the Search Text field you must type something in (a string) and then in the next field you can choose whether or not you want the query to use ‘LIKE’ (contains) or not (‘=’) this obvious limits your query strength.. Note that you can only query string fields here - don’t even bother trying to query a numeric field…Even if you specify it in the ‘Search Fields’ field. My initial thought was to put OR and specify the SITUSSTATE field…turns out that field is left null in some cases… So how ‘bout something loose? here we’ll select every record that contains an A in any of it’s fields…that ought a cover our bases on that limitation.

Moving on to the Layer Definitions field, here is where you can add some special sauce: Note that I can actually specify a like statement and quotes on the string and where I want my wildcards…I could use numerics here as well, less than, greater than etc. Feasibly to extract all data from a service with a 1000 record limit we would iterate through OID < 1000, OID >999 AND OID<2000, OID > 1999 AND OID <3000 etc.etc.

Helpful GP Environment enumeration

This tip is worth 5hrs of my life…

Always, always, always remember to set the GP workspace and scratch workspace when running a gp process in an SOE.

Handy little snippet to see what your geoprocessing environment values are when running in black box mode.

string envsList = "";
IGpEnumList baz;
baz = gp.ListEnvironments("*");
string envs = baz.Next();
while (envs != "")
{
envsList += envs + "::" + gp.GetEnvironmentValue(envs) + Environment.NewLine;
envs = baz.Next();
}

Failed to Access IIS Metabase

Yellow screen of death =

Description: An unhandled exception occurred during the execution of the current web request. Please review the stack trace for more information about the error and where it originated in the code.

Exception Details: System.Web.Hosting.HostingEnvironmentException: Failed to access IIS metabase.

The process account used to run ASP.NET must have read access to the IIS metabase (e.g. IIS://servername/W3SVC). For information on modifying metabase permissions, please see http://support.microsoft.com/?kbid=267904.

Checked permissions on this dir and ASPNET had read access but this fixed it.

I just ran into this issue when opening manager after installing ArcGIS Server 10 on a new XP box and this fixed my problem.

1. Grant the local ASPNET user rights to C:\WINDOWS\Microsoft.NET\Framework\v2.0.50727\Temporary ASP.NET Files

2. At the CMD run this C:\WINDOWS\Microsoft.NET\Framework\v2.0.50727\aspnet_regiis.exe -ga ASPNET

3.At the CMD run this iisreset

I’m betting IIS just needed to be reset.
